# FixIt — הסברים טכניים להצגה

מסמך הכנה לפגישת ביניים עם המרצה.  
כולל: מה כל טכנולוגיה, למה בחרנו בה, ומה האלטרנטיבה שלא בחרנו.

---

## מה זה FixIt?

FixIt היא פלטפורמת שירותים מקומית המחברת בין **מבקשי שירות** (Requesters) ל**מתקנים** (Fixers) — אנשים שרוצים שיעזרו להם עם עבודות קטנות כמו הרכבת רהיטים, תלייה, צביעה, אינסטלציה — ובין אנשי מקצוע מהאזור שמוכנים לבצע אותן.

**המנגנון המרכזי:** הדרישה מתפרסמת, ה-Fixers מגישים הצעות מחיר — Requester בוחר את הטוב ביותר. מודל תחרותי ושקוף, בניגוד לוואטסאפ שכונתי שהוא הפתרון הנוכחי בישראל.

**הפלטפורמה:** אפליקציה לנייד (iOS + Android) ואתר אינטרנט — מאותו קוד בדיוק.

## טכנולוגיות צד לקוח (Frontend)

### React Native + Expo

**מה זה:**  
React Native היא ספרייה של Meta (פייסבוק) שמאפשרת לכתוב אפליקציה אחת בשפת JavaScript/TypeScript, ולהריץ אותה גם על iOS וגם על Android — בלי לכתוב קוד פעמיים.  
Expo היא שכבה נוספת מעל React Native שמפשטת את הפיתוח: מנהלת את ה-build, מספקת גישה לחומרה (מצלמה, מיקום, התראות), ומאפשרת גם הרצה בדפדפן.

**למה בחרנו:**  
רצינו שהאפליקציה תעבוד גם על נייד וגם בדפדפן — מאותו קוד. React Native + Expo מאפשרים בדיוק את זה. בנוסף, זו הטכנולוגיה המובילה בתעשייה ל-cross-platform mobile, ומוכרת לנו מקורסים קודמים.

**האלטרנטיבה שלא בחרנו:**  
Flutter (של גוגל) — גם cross-platform, אבל בשפת Dart שאף אחד מאיתנו לא מכיר. SwiftUI/Kotlin לנייד נייטיב — היה מחייב שתי אפליקציות נפרדות.

### TypeScript

**מה זה:**  
TypeScript היא JavaScript עם הגדרות טיפוסים (types). כלומר, במקום `let price = getPrice()` כותבים `let price: number = getPrice()`, ואם בטעות מנסים להכניס לשם טקסט — הקוד לא יקמפל. זה תופס המון שגיאות עוד לפני שמריצים את הקוד.

**למה בחרנו:**  
הפרויקט גדול, עם שלושה מפתחים — TypeScript מונע שגיאות שנובעות מאי-הבנה בין חברי הצוות. בנוסף, ה-IDE מספק autocomplete מלא, מה שמאיץ פיתוח משמעותית.

**האלטרנטיבה שלא בחרנו:**  
JavaScript רגיל — פשוט יותר, אבל בפרויקטים גדולים שגיאות טיפוסים הן אחת הסיבות הנפוצות לבאגים. עם TypeScript end-to-end (גם backend וגם frontend) יש משמעת משותפת לכל הצוות.

### Google Maps API

**מה זה:**  
שירות מפות של גוגל. בפרויקט משתמשים בשני חלקים:
- **Maps JavaScript API** — להצגת מפה אינטראקטיבית בדפדפן עם pins של משימות
- **Places API** — להשלמה אוטומטית של כתובות כשמזינים מיקום למשימה (autocomplete)

**למה בחרנו:**  
Google Maps היא הסטנדרט בתעשייה, בעיקר בישראל (Waze, כמעט כל אפליקציה ישראלית). מדויקת לרחובות ישראליים, ויש לה תמיכה טובה ל-React Native.

**האלטרנטיבה שלא בחרנו:**  
OpenStreetMap (חינמי לגמרי) — פחות מדויק לישראל ופחות תמיכה ל-React Native. Mapbox — יותר גמיש ויפה, אבל יותר יקר ופחות מוכר.

### Firebase Authentication + Firebase Storage

**מה זה:**  
Firebase הוא שירות ענן של גוגל. אנחנו משתמשים בשני חלקיו:

- **Firebase Authentication** — מנהל הרשמה, התחברות, שחזור סיסמה, ואימות אימייל. המשתמש מתחבר בצד הלקוח, ו-Firebase מחזיר **ID Token** — אישור מוצפן שמוכיח שהוא המשתמש הזה. הצד השרת מקבל את הטוקן הזה ומאמת אותו.

- **Firebase Storage** — אחסון קבצים בענן. כשמשתמש מעלה תמונה של משימה, הקובץ עולה ישירות מהאפליקציה ל-Firebase Storage, ואנחנו שומרים רק את הכתובת (URL) במסד הנתונים שלנו.

**למה בחרנו:**  
הייתה לנו אפשרות לנהל את ה-auth בעצמנו (שמירת סיסמאות, הנפקת JWT tokens...) — אבל זה מורכב ומסוכן. Firebase Auth מבצע את הכל בצורה מאובטחת, והסיסמאות לא נוגעות בשרת שלנו בכלל. בנוסף, Firebase Storage חוסך פיתוח שרת העלאת קבצים.

**האלטרנטיבה שלא בחרנו:**  
Auth0 — שירות דומה אבל יקר יותר. ניהול auth עצמי (bcrypt + JWT) — אפשרי אבל דורש הרבה זהירות אבטחתית.

## טכנולוגיות צד שרת (Backend)

### Node.js + Express

**מה זה:**  
- **Node.js** — סביבת ריצה שמאפשרת להריץ JavaScript/TypeScript על השרת (לא רק בדפדפן).
- **Express** — ספריית web קטנה שיושבת מעל Node.js ומאפשרת להגדיר API endpoints בקלות. למשל: "כשמגיעה בקשת POST לכתובת /api/tasks — הרץ את הפונקציה הזאת".

**למה בחרנו:**  
TypeScript end-to-end — frontend ו-backend באותה שפה. כל הצוות מכיר JavaScript. Express קל ומהיר לפיתוח ומאוד נפוץ בתעשייה.

**האלטרנטיבה שלא בחרנו:**  
Python + Django/FastAPI — שפה שגם מכירים, אבל היינו מאבדים את היתרון של TypeScript משותף. NestJS — framework יותר structured מעל Node.js, אבל overhead גבוה יותר לפרויקט בסדר גודל שלנו.

---

### PostgreSQL + PostGIS

**מה זה:**  
- **PostgreSQL** — מסד נתונים רלציוני (כמו MySQL), בקוד פתוח. שומר נתונים בטבלאות עם קשרים בין הטבלאות.
- **PostGIS** — תוסף ל-PostgreSQL שמוסיף תמיכה בנתונים גאוגרפיים. כלומר, אפשר לשמור **נקודות על המפה** (קווי אורך ורוחב) בעמודה מיוחדת, ואז לבצע שאילתות כמו: "תן לי את כל המשימות שנמצאות ברדיוס של 10 ק"מ מהמיקום שלי".

**בפרויקט:**  
כל משימה שמתפרסמת מאוחסנת עם מיקום גאוגרפי. כשה-Fixer פותח את מפת הגילוי, הוא מקבל את כל המשימות הפתוחות סביבו — זה נעשה עם שאילתת `ST_DWithin` של PostGIS, שמחזירה רק משימות בתוך הרדיוס שנבחר.

**למה בחרנו:**  
PostgreSQL הוא הסטנדרט לפרויקטים שדורשים מסד נתונים יציב ומורכב. PostGIS הוא הפתרון המקצועי לשאילתות גאוגרפיות — שימוש בו הוא תשובה CS נכונה, לעומת האלטרנטיבה של לאחסן latitude/longitude כשני מספרים ולעשות את חישוב המרחק בקוד (פחות יעיל ופחות מדויק).

**האלטרנטיבה שלא בחרנו:**  
MongoDB — מסד נתונים לא-רלציוני (NoSQL), יש לו גם תמיכה גאוגרפית, אבל הפרויקט שלנו עם קשרים בין Users, Tasks, Bids, Reviews מתאים יותר למבנה רלציוני. MySQL — דומה ל-PostgreSQL אבל PostGIS זמין רק ל-PostgreSQL.

---

### Prisma ORM

**מה זה:**  
ORM (Object-Relational Mapper) הוא שכבה בין הקוד למסד הנתונים. במקום לכתוב SQL גולמי כמו `SELECT * FROM tasks WHERE id = '...'`, כותבים קוד TypeScript רגיל: `prisma.task.findUnique({ where: { id } })`.

Prisma מייצר גם **TypeScript types** אוטומטית מה-schema של מסד הנתונים — כך שאם שינינו עמודה ב-DB, הקוד לא יקמפל עד שנתקן גם את כל השימושים בה.

**בפרויקט:**  
ה-schema של Prisma מגדיר את כל הטבלאות (User, Task, Bid, Review, Message, Notification, PortfolioItem). השינויים ב-schema מנוהלים דרך **migrations** — קבצים שמתעדים את שינויי מסד הנתונים לאורך הזמן, ומאפשרים לכל חבר בצוות לסנכרן את ה-DB המקומי שלו.

**למה בחרנו:**  
Prisma הוא הכי פופולרי ב-Node.js ecosystem כיום. הייחודיות שלו היא ה-type safety — שגיאות ב-queries נתפסות בזמן קומפייל ולא בזמן ריצה.

**האלטרנטיבה שלא בחרנו:**  
Sequelize — ORM ותיק יותר ל-Node.js, אבל type safety חלשה יותר. TypeORM — דומה ל-Prisma, פחות נוח. SQL גולמי (raw queries) — שליטה מלאה, אבל הרבה יותר verbose וקל לטעויות.

---

### Zod

**מה זה:**  
ספרייה לvalidation של נתונים ב-TypeScript. מגדירים schema שמתאר מה צריך להיכנס לנקודת קצה (endpoint), ו-Zod בודק שהנתונים שמגיעים מהלקוח תקינים לפני שמבצעים כל פעולה.

**בפרויקט:**  
לדוגמה, לפני שיוצרים משימה חדשה, Zod מוודא שיש כותרת (לא ריקה, מקסימום 80 תווים), יש קטגוריה מהרשימה המותרת, יש מחיר חיובי או שהמחיר הוא "Quote Required" וכו'. אם משהו לא תקין — מוחזרת שגיאה ברורה עם פירוט מה חסר.

**למה בחרנו:**  
בלי validation, כל משתמש יכול לשלוח כל דבר לשרת. Zod הוא הפתרון הסטנדרטי ב-TypeScript ecosystem, ומשתלב ישירות עם TypeScript types.

**האלטרנטיבה שלא בחרנו:**  
Joi — ספריית validation ותיקה יותר, אבל פחות TypeScript-native. validation ידני — אפשרי אבל מייצר המון קוד חוזר שקל לפספס.

---

### Socket.io

**מה זה:**  
ספרייה לתקשורת real-time בין לקוח לשרת. בניגוד ל-HTTP רגיל שבו הלקוח שולח בקשה והשרת מחזיר תשובה, Socket.io שומר **חיבור פתוח** (WebSocket) כך שהשרת יכול לשלוח הודעות ללקוח בכל רגע — בלי שהלקוח ישאל.

**בפרויקט:**  
הצ'אט בין Requester ל-Fixer (אחרי שהצעה התקבלה) עובד דרך Socket.io. כל משימה מקבלת "חדר" משלה (`task_chat_{id}`), ושני הצדדים נכנסים לאותו חדר. כשמישהו שולח הודעה — כולם בחדר מקבלים אותה מיד.  
אם הצד השני לא מחובר — במקום WebSocket, מישלחים לו **push notification** לנייד.

**למה בחרנו:**  
Socket.io הוא הסטנדרט לצ'אט real-time ב-Node.js. מטפל אוטומטית ב-fallback (אם WebSocket לא נתמך — עובר ל-polling), ומנהל חדרים ו-events בצורה נוחה.

**האלטרנטיבה שלא בחרנו:**  
WebSocket גולמי — אפשרי, אבל Socket.io מוסיף הרבה ניהול שהיינו צריכים לממש לבד. Server-Sent Events (SSE) — תמיכה רק בכיוון אחד (שרת לקוח), לא מתאים לצ'אט דו-כיווני.

---

### Expo Push Notifications

**מה זה:**  
שירות של Expo לשליחת push notifications לאפליקציות נייד. כל מכשיר מקבל **Expo Push Token** — מזהה ייחודי. כשרוצים לשלוח notification, שולחים בקשה לשרתי Expo עם הטוקן והתוכן, והם מעבירים את ה-notification למכשיר.

**בפרויקט:**  
אחרי כל פעולה משמעותית — הגשת הצעה, אישור הצעה, דחיית הצעה, הודעת צ'אט — נשלחת push notification לצד הרלוונטי. ה-token נשמר ב-DB כשהמשתמש נכנס לאפליקציה בפעם הראשונה.

**למה בחרנו:**  
הייתה אפשרות לעבוד ישירות עם APNs (Apple) ו-FCM (Google) — אבל כל אחד מהם דורש הגדרה נפרדת ו-SDK נפרד. Expo מאחד את שניהם לממשק אחד פשוט, מה שחוסך הרבה זמן הגדרה.

**האלטרנטיבה שלא בחרנו:**  
Firebase Cloud Messaging ישירות — יותר שליטה, אבל יותר עבודה. OneSignal — שירות דומה של צד שלישי.

## תשתית (Infrastructure)

### Docker

**מה זה:**  
כלי שמאפשר להריץ תוכנה בתוך **container** — סביבה מבודדת שמכילה הכל: את התוכנה, את ספריות שהיא צריכה, ואת ההגדרות. ה-container הזה רץ אותו דבר על כל מחשב.

**בפרויקט:**  
השתמשנו ב-Docker כדי להריץ את PostgreSQL + PostGIS מקומית. כל חבר בצוות מריץ `docker compose up` ומקבל מסד נתונים זהה — בלי להתקין PostgreSQL ידנית, ובלי בעיות "אצלי עובד ואצלך לא".

**למה בחרנו:**  
PostGIS הוא תוסף ל-PostgreSQL שקצת מסובך להתקין ידנית. עם Docker, כולם מקבלים image מוכן עם PostGIS כבר בפנים (`postgis/postgis:15-3.3`). אחידות סביבת הפיתוח חוסכת שעות של debugging.

**האלטרנטיבה שלא בחרנו:**  
התקנה ידנית של PostgreSQL על כל מחשב — אפשרי, אבל גרסאות שונות ובעיות התקנה גורמות לסיוטים.

---

### GitHub Actions CI (Continuous Integration)

**מה זה:**  
CI (Continuous Integration) הוא תהליך אוטומטי שרץ בכל פעם שמישהו פותח Pull Request ב-GitHub. הוא מריץ בדיקות אוטומטיות (אצלנו: TypeScript typecheck + ESLint) ומדווח אם הכל עבר.

**בפרויקט:**  
יש לנו שני workflows — אחד ל-backend ואחד ל-frontend. בכל PR, GitHub מריץ את שניהם. אם יש שגיאת TypeScript שמישהו פיספס — הCIתופס אותה לפני שהקוד מוזג ל-main.

**למה בחרנו:**  
עם שלושה מפתחים עובדים על branches נפרדות, קל לשבור קוד בטעות. CI הוא רשת הגנה שמבטיחה שה-main branch תמיד קומפיילבל.

**האלטרנטיבה שלא בחרנו:**  
בדיקה ידנית לפני merge — לא סקיילבל ותלויה בזיכרון. CircleCI/Jenkins — כלי CI אחרים, אבל GitHub Actions משולב ישירות ב-GitHub ובחינם לפרויקטים ציבוריים.

## החלטות ארכיטקטוראליות מעניינות

### 1. Monorepo — קוד אחד, שני פרויקטים

כל הקוד (backend + frontend) נמצא ב-repository אחד, עם npm workspaces שמנהל את שני הפרויקטים. זה מאפשר לכל חבר צוות לראות את כל הפרויקט, לשתף הגדרות (TypeScript, ESLint), ולוודא שגרסאות ספריות מסונכרנות.

### 2. פרטיות מיקום

כשמשימה מתפרסמת, ה-Fixer רואה רק **אזור כללי** (pin על המפה ברמת שכונה) — לא את הכתובת המדויקת. הכתובת המדויקת נחשפת רק לאחר שהצעתו התקבלה. זה יישום של עיקרון minimal data exposure.

בנוסף, ה-pin על המפה מוזז מספר מטרים אקראיים (אבל דטרמיניסטיים — תמיד אותה הזזה לאותה משימה) כדי שלא יהיה אפשר לנחש את הכתובת מהמפה.

### 3. Atomic Transaction לאישור הצעה

כשRequester מאשר הצעה, צריכות לקרות כמה פעולות בו זמנית: המשימה עוברת ל-IN_PROGRESS, ההצעה המאושרת עוברת ל-ACCEPTED, וכל ההצעות האחרות עוברות ל-REJECTED. אם אחת מהפעולות נכשלת באמצע — הנתונים יהיו לא עקביים.

הפתרון: Prisma transaction — כל הפעולות מתבצעות ביחד, ואם אחת נכשלת, הכל מתבטל. זה גם מטפל בבעיית race condition: אם שני Requesters ינסו לאשר בו זמנית (לא רלוונטי אצלנו, אבל תיאורטית) — רק אחד יצליח.
