FixIt
A Location-Aware Task Marketplace
Guy Stein · Guy Zilberstein · Guy Shick
CS Final Project · Spring 2026
Opening: One sentence intro — "FixIt is a task marketplace connecting people who need small jobs done with local professionals who bid on them." Then move straight to the problem slide.
The Problem

Finding local help is broken.

🔨
Contractors won't take small jobs.

Licensed tradespeople ignore micro-gigs — mounting a TV, assembling IKEA furniture, fixing a leaky tap.

💸
Prices are opaque.

You pay whatever they quote — no transparency, no comparison, no market rate.

🤷
No accountability.

The current solution is a WhatsApp neighborhood group — unstructured, unreliable, no trust signals.

Context: Name the Israeli reality — WhatsApp groups like "שכונת X – עזרה הדדית" are the actual incumbent. This is a real, everyday friction point in Israel. The market exists; it just has no infrastructure.
The Solution

A transparent, competitive bidding marketplace.

1

Post a Task

Title · photos · category · budget · location pin

2

Fixers Bid

Local professionals submit price + pitch (max 15 bids)

3

Accept & Chat

Pick the best offer — real-time chat opens instantly

4

Complete & Review

Mark done · pay via Bit/Paybox · leave a star rating

🔄 One account, two roles
Users switch between Requester and Fixer mode in the top nav — same account, no re-registration.
📱 Mobile + Web, one codebase
React Native + Expo — same TypeScript runs natively on iOS, Android, and browser with full feature parity.
Key angle: The Requester sets terms, Fixers compete — inverts the usual power dynamic. Highlight the 15-bid cap as a deliberate UX constraint: creates urgency for Fixers, prevents notification overload for Requesters. Mention the unified account design — many users are both a Requester for their home and a Fixer professionally.

Backend
🟢 Node.js 🟦 TypeScript 🐘 PostgreSQL 📍 PostGIS ▲ Prisma ORM ⚡ Socket.io ✅ Zod 🔥 Firebase Admin

PostGIS extends PostgreSQL with spatial types and a GiST index — enables radius-based discovery in a single SQL predicate instead of app-level coordinate math.

Firebase Admin verifies ID tokens server-side. Passwords never touch our backend.

Points to stress:
(1) PostGIS was a deliberate choice — not just storing lat/lng as floats. You get a proper spatial index (GiST), OGC-standard functions, and correct geodetic distance. Mention you could have done bounding-box filtering in application code, but ST_DWithin with a spatial index is the correct CS answer.
(2) Firebase Admin = zero password handling on the backend. A security win by design, not just convenience.
(3) TypeScript end-to-end — both frontend and backend share the same type discipline despite being separate packages in a monorepo with npm workspaces.

Walk through three request paths:
(1) REST: Axios with Bearer token → authMiddleware verifies via Firebase Admin SDK → route handler → Prisma → PostgreSQL.
(2) Real-time: Socket.io handshake carries the same Firebase ID token — same trust boundary, no second auth system.
(3) Push: notificationService always writes a Notification row to DB first (so in-app Notification Center always works), then sends Expo push only if the user has a registered token. Non-blocking — a missing token doesn't throw.

Key design point: Firebase Admin is the single identity source. We never issue our own tokens — we piggyback on Google's token infrastructure.

Under the Hood

CS-Interesting Decisions

📍

Geospatial Queries

PostGIS ST_DWithin on a GiST-indexed geometry column — radius discovery in a single SQL predicate. Privacy: markers offset 10–50m via deterministic hash of task ID.

-- radius discovery
WHERE ST_DWithin(
  coordinates,
  ST_SetSRID(ST_MakePoint(
    :lng, :lat), 4326),
  :radiusMeters
)
⚖️

Atomic Bid Acceptance

Accepting one bid runs a Prisma transaction: task → IN_PROGRESS, winner → ACCEPTED, all other pending → REJECTED in one atomic DB write. Race-condition safe.

prisma.$transaction([
  task.update(IN_PROGRESS),
  bid.update(ACCEPTED),
  bid.updateMany(REJECTED)
])
⚡

Real-Time + Offline Fallback

Socket.io rooms scoped per task — both parties join task_chat_{id}. If recipient is offline, notificationService sends an Expo push. Same Firebase token authenticates the WS handshake.

// offline fallback
if (!recipientInRoom) {
  sendNotification(
    userId, NEW_MESSAGE
  )
}
Per card:
(1) PostGIS — you could store lat/lng as two floats and do bounding-box math in app code. The correct CS answer is a spatial index. Also: location masking is a product privacy requirement solved with deterministic pseudorandomness — same offset every render (hash of task ID), but not recoverable to the real address.
(2) Transaction — without it, two concurrent Accept requests could both read "PENDING" and both commit, leaving two accepted bids. The Prisma transaction makes the accept atomic at the DB level — whichever request commits first wins, the second gets a conflict error.
(3) Real-time — same Firebase ID token works for both REST (Authorization header) and WebSocket (handshake auth object). No second auth system.

Dynamic pricing uses historical bids + PostGIS — no new infrastructure. NLP tags require only the review text we already store.

Likely questions:
• "Why Firebase Auth instead of your own JWT?" — Security best practice: delegate auth to a proven provider, verify tokens server-side with Admin SDK. We never handle passwords.
• "Why PostGIS instead of storing lat/lng?" — Spatial index (GiST), proper geodetic distance, OGC-standard functions. Scales correctly as the dataset grows.
• "How does the transaction prevent race conditions?" — Two concurrent Accept requests compete at the DB commit. Only one can win; the second gets a constraint violation. Without the transaction, both could read "PENDING" and both commit.
• "Biggest technical challenge?" — Dual-platform maps: Google Maps JS API (web) vs react-native-maps (mobile) have different APIs entirely, requiring two map implementations sharing the same data layer and filter state.
• "Why Expo over bare React Native?" — Unified dev tooling, OTA updates, built-in wrappers for location/camera/push. We can eject to bare if we ever need a fully custom native module.
