FixIt
A Location-Aware Task Marketplace
Guy Stein · Guy Zilberstein · Guy Shick
CS Final Project · Spring 2026
פתיחה: משפט אחד — "FixIt הוא שוק משימות עם מודעות מיקום שמחבר בין אנשים שצריכים עזרה לבין אנשי מקצוע מקומיים שמתמחרים על הביצוע." עוברים מיד לשקף הבא.
The Problem

Finding local help is broken.

🔨
Contractors won't take small jobs.

Licensed tradespeople ignore micro-gigs — mounting a TV, assembling IKEA furniture, fixing a leaky tap.

💸
Prices are opaque.

You pay whatever they quote — no transparency, no comparison, no market rate.

🤷
No accountability.

The current solution is a WhatsApp neighborhood group — unstructured, unreliable, no trust signals.

הקשר: תזכירו את המציאות הישראלית — קבוצות וואטסאפ כמו "שכונת X – עזרה הדדית" הן הפתרון הנוכחי בפועל. אין מחירים שקופים, אין ביקורות, אין מבנה. הדגישו: אנחנו לא ממציאים שוק חדש — אנחנו נותנים לשוק שכבר קיים תשתית מסודרת.
The Solution

A transparent, competitive bidding marketplace.

1

Post a Task

Title · photos · category · budget · location pin

2

Fixers Bid

Local professionals submit price + pitch (max 15 bids)

3

Accept & Chat

Pick the best offer — real-time chat opens instantly

4

Complete & Review

Mark done · pay via Bit/Paybox · leave a star rating

🔄 One account, two roles
Users switch between Requester and Fixer mode in the top nav — same account, no re-registration.
📱 Mobile + Web, one codebase
React Native + Expo — same TypeScript runs natively on iOS, Android, and browser.
זווית מרכזית: הדורש מפרסם ומקבל הצעות — כוח השוק עובד לטובתו. מגבלת 15 ההצעות היא החלטת UX מכוונת: יוצרת לחץ על המתקנים לבוא עם הצעה רצינית, ומונעת הצפת התראות לדורש. חשבון אחד לשני תפקידים — הרבה אנשים הם גם דורשים בבית וגם מתקנים במקצועם.

הסבר רשימת הפיצ'רים:

מה שבנינו: הדגישו שכל הזרימה המרכזית עובדת end-to-end — דורש מפרסם משימה עם תמונות ומיקום, מתקן מגלה אותה במפה, מגיש הצעת מחיר, הדורש מקבל, ויש ניווט בזמן אמת לכתובת.

Push Notifications — עובד: הדורש מקבל התראה כשמגיעה הצעה, המתקן מקבל התראה כשההצעה התקבלה/נדחתה.

המלצות אישיות: שני סוגים — (1) לדורש: כשמפרסם משימה, המערכת תוכל להציע מתקנים מומלצים לפי קטגוריה + קרבה + דירוג. (2) למתקן: משימות מומלצות בדף הגילוי לפי ההתמחויות שהגדיר. הנתונים שנצטרך כבר נמצאים ב-DB — specializations, ratings, locations.

Dynamic pricing — אגרגציה על הצעות מחיר היסטוריות מאותה קטגוריה ואזור. הנתונים כבר נאספים — רק צריך לחשב.
Stack

Tech Stack

Frontend
⚛ React Native 📱 Expo SDK 54 🟦 TypeScript 🗺 Google Maps 🔥 Firebase Auth

Same codebase → iOS · Android · Web. Firebase client handles the full auth lifecycle — no custom session logic.

Infrastructure
🐳 Docker ⚙️ GitHub Actions CI 🔔 Expo Push ☁️ Firebase Storage

Lint + typecheck on every PR (separate backend & frontend CI workflows).

Backend
🟢 Node.js + Express 🟦 TypeScript 🐘 PostgreSQL 📍 PostGIS ▲ Prisma ORM ⚡ Socket.io ✅ Zod 🔥 Firebase Admin

PostGIS extends PostgreSQL with spatial types and a GiST index — enables radius-based discovery in a single SQL predicate.

Firebase Admin verifies ID tokens server-side. Passwords never touch our backend.

הסבר כל טכנולוגיה — צד הלקוח:
• React Native — ספרייה של מטא לפיתוח אפליקציות מובייל. כותבים TypeScript אחד שמתרגם לרכיבים מקומיים של iOS ו-Android.
• Expo SDK — מסגרת עבודה מעל React Native. מוסיפה OTA updates (עדכונים ללא פרסום חדש בחנות), ו-SDK מוכן למצלמה, מיקום ו-Push Notifications.
• TypeScript — JavaScript עם טיפוסים סטטיים. כמו Java אבל לווב. מקטין באגים בזמן כתיבת הקוד.
• Google Maps API — שירות מפות של גוגל. Geocoding (כתובת לקואורדינטות), תצוגת מפה ומרחקים.
• Firebase Auth — שירות אימות משתמשים של גוגל. מטפל בהרשמה, התחברות ושחזור סיסמה. מחזיר JWT Token — מחרוזת מוצפנת שמוכיחה זהות המשתמש.

צד השרת:
• Node.js + Express — Node.js היא סביבת ריצה ל-JavaScript בצד השרת. Express היא פריימוורק HTTP שמאפשרת להגדיר Routes ו-Middleware בקלות.
• PostgreSQL — בסיס נתונים רלציוני חזק ופופולרי. בחרנו אותו בגלל תמיכת PostGIS.
• PostGIS — הרחבה ל-PostgreSQL שמוסיפה סוגי נתונים גיאוגרפיים ו-Spatial Index (GiST). מאפשרת שאילתה כמו "תחזיר כל משימות ברדיוס 10 ק"מ ממני" בשורת SQL אחת.
• Prisma ORM — כלי שמגשר בין TypeScript לבין PostgreSQL. מגדירים Schema בסינטקס מיוחד, Prisma מייצר טיפוסי TypeScript אוטומטית — אין צורך בכתיבת SQL ידנית.
• Socket.io — ספרייה לתקשורת דו-כיוונית בזמן אמת (WebSocket). תשמש לצ'אט בין דורש למתקן.
• Zod — ספרייה לוולידציה של קלט. מוודאת שמה שהמשתמש שולח ל-API תואם את המבנה הצפוי.
• Firebase Admin SDK — גרסת השרת של Firebase. מאפשרת לוודא שה-JWT Token שהלקוח שלח אמיתי ולא מזויף.

תשתית:
• Docker — כלי Containerization שמארז את הסביבה כך שכל מפתח עובד באותה סביבה בדיוק.
• GitHub Actions CI — אוטומציה שרצה על כל PR — בודקת Lint ו-TypeCheck ומונעת קוד שבור מלהתמזג.
• Expo Push Notifications — שירות של Expo לשליחת התראות Push למכשיר נייד ברקע.
• Firebase Storage — שירות אחסון קבצים של גוגל. המשתמש מעלה תמונות ישירות מהמכשיר — אנחנו שומרים רק את ה-URL.

הסבר שלוש השכבות:

• Client (לקוח) — אפליקציית Expo שרצה על iOS, Android ובדפדפן מאותו קוד. כאן המשתמש מתחבר דרך Firebase, מעלה תמונות ישירות ל-Firebase Storage (בלי לעבור דרך השרת שלנו), ושולח בקשות API לשרת.

• Backend (שרת) — שרת Express שמקבל בקשות, מאמת זהות, מריץ לוגיקה עסקית ומחזיר נתונים. כאן יושב גם Socket.io לצ'אט בזמן אמת, ושירות ההתראות שכותב ל-DB ושולח Push Notification.

• Data (נתונים) — PostgreSQL עם PostGIS לכל נתוני האפליקציה. Prisma מתרגם TypeScript לשאילתות SQL. ה-GiST Index מאפשר שאילתות גיאוגרפיות מהירות.

Firebase הוא שירות חיצוני של גוגל שמטפל לנו בשתי בעיות — Auth ו-Storage — בלי שנצטרך לבנות את זה מאפס.
Architecture — 2 of 2

How a Request Flows

1
User logs in → Firebase Auth returns a signed JWT Token (valid 1 hour)
↓
2
App makes API call — Axios Interceptor automatically attaches the token to every request header
↓
3
authMiddleware on the server verifies the token via Firebase Admin SDK — confirms identity without storing any password
↓
4
Route handler runs business logic — Zod validates input, authorization rules are checked
↓
5
Prisma ORM translates to SQL → PostgreSQL + PostGIS executes (spatial index used for location queries)
↓
6
Response returns to client — if the action triggers a notification, notificationService writes to DB and fires Expo Push
עברו שלב-שלב על הזרימה:

1. המשתמש מתחבר → Firebase מחזיר JWT Token — מחרוזת מוצפנת עם פרטי הזהות, תקפה לשעה.
2. כל בקשת API שיוצאת מהאפליקציה — Axios Interceptor מוסיף אוטומטית את ה-Token כ-Authorization Header. אף מסך לא צריך לטפל בזה ידנית.
3. authMiddleware בשרת מקבל את ה-Token, שולח אותו ל-Firebase Admin לאימות, ומוסיף את פרטי המשתמש ל-Request Object. אם ה-Token לא תקין — 401.
4. ה-Route Handler מריץ לוגיקה — Zod בודק שגוף הבקשה תקין, ומבצעים בדיקות הרשאה (למשל: רק הדורש שיצר משימה יכול לקבל הצעות עליה).
5. Prisma מתרגם ל-SQL. בשאילתות גיאוגרפיות — PostGIS ו-GiST Index נכנסים לפעולה.
6. תשובה חוזרת. אם האירוע מחייב התראה — notificationService כותב שורה ל-DB וגם שולח Push Notification למכשיר.

נקודה חשובה: הסיסמאות של המשתמשים לא נוגעות בשרת שלנו בכלל.
